Skill Repo Evolution

Routing first, formats second, recipes underneath.

The repo already has strong ingredients: a fat core skill, narrow commands, references, templates, and export scripts. The next move is to make routing explicit. We will treat output surfaces like PDF, MP4, MOV, and WebM as first-class decisions, then hang video recipes and export commands off that resolver instead of burying them behind flags.

Branch

`codex/skill-repo-resolver-plan`

Dedicated branch for the routing and format architecture work.

North Star

Legible skill system

Every capability is reachable, every format has a clear contract, every prompt shape maps to a known path.

Why Change

Today, video and export behavior is technically capable but conceptually compressed. `generate-video`, `render-video`, `--pdf`, `--poster-export`, and format choices are mixed together. The result works, but the routing layer is implicit.

Formats hidden as flags No explicit output resolver Prompting vocabulary under-modeled

Target State

A resolver-driven system. User intent picks an output family first, then the right command, then the right recipe, then the right references. Skills stop inventing format logic ad hoc.

Resolver.md Output contracts Recipe manifests

Immediate Outcome

Clearer commands, cleaner docs, and a repo that can grow into more delivery surfaces without becoming a pile of special cases.

PDF becomes first-class Video recipes expand safely Reachability gets test coverage

Execution Flow

1. Resolver
Define routing

Write one authoritative routing table for output family, command selection, and required context.

`output-resolver.md`
Intent: read / present / share / edit / embed
Input shapes: topic / doc / deck / HTML / frames
2. Manifest
Make it machine-readable

Add a manifest for commands, formats, cost tiers, prerequisites, and references.

Command ids + aliases
Supported formats
Required refs + prerequisite tools
3. Surfaces
Split export surfaces

Promote PDF and asset export to first-class concepts instead of leaving them as incidental flags.

`/export-pdf`
`/export-assets`
Keep flags as compatibility aliases
4. Video recipes
Model delivery intent

Keep the core commands, but route them to recipe-specific flows like reel, overlay, browser loop, and long-form explainer.

Long-form explainer
Social reel
Transparent overlay
5. Evals
Test reachability

Add resolver evals and a `check-resolvable` pass so every command and recipe is discoverable.

Prompt routing fixtures
Manifest drift checks
Unreachable capability detector

Prompting Upgrade For HyperFrames

We will mirror HyperFrames’ current prompting model directly in local references so the repo can reason about more than just `style=reel`. The key move is to encode prompt shape and vocabulary, not just technical flags.

Prompt shapes: cold start vs warm start
Vocabulary: motion feel, caption tone, transition energy, voice, quality
Iteration: small editor-style follow-ups, not full re-prompts

Guiding Rule

We are not trying to add “more commands because more commands feel powerful.” We are trying to make the organization legible. The fat skill remains the domain brain. Commands stay narrow. References load on demand. Recipes handle variation underneath. The resolver becomes the management layer that tells the system which path to take.

Next implementation step after review: add the resolver/reference files first, then thin the existing commands around them.
