01 · Opening

The dashboard said throughput was fine. The queue depth graph said otherwise.

Between the two, we lost three customers before anyone thought to reconcile the gap. This dispatch is a post-mortem written while the traces are still warm.

The quiet finding

The ceiling wasn't in the code. It was in the partition key.

8.4×
throughput in production
Before
500 rps

Ceiling — not the average. Past this point the queue depth climbed linearly until workers OOM'd and the on-call phone rang at 03:14.

After
4,200 rps

Same hardware, same binaries, one topic config change. Partition by tenant, not by message type.

Reframe

Partition keys encode your real contract.

Everything else is commentary. When tenant ordering is the guarantee you actually sell, partition by tenant. When message type is, partition by type. We had confused the two for eleven months.

Latency

P99 down 68%.

No head-of-line blocking from oversize payloads.

Throughput

8.4× ceiling.

Same hardware, no scaling event.

Cost

No change.

The fix was a partition key, not more boxes.

Regressions

Zero so far.

Tenant ordering was already our contract — partitioning by it is lossless.
